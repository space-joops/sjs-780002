### 왜 게임 상태를 `useState`에 넣지 않았을까?

이게 이 코드에서 가장 중요한 설계 결정입니다.

```ts
// ❌ 이렇게 짜면 안 되는 이유
const [junks, setJunks] = useState<Junk[]>([]);
// 매 프레임 setJunks(...) → 초당 60번 React 리렌더 → 60번 diff/reconcile
// 쓰레기 30개면 프레임마다 배열 30개 새로 만들고, VDOM 훑고… 폰에서 즉사
```

```ts
// ✅ 실제 코드 (56~90행)
let score = 0;
let lives = 3;
let junks: Junk[] = [];
const player = { x: 0, y: 0, /* … */ };
```

`useEffect(() => { … }, [])`의 콜백은 **딱 한 번** 실행되고, 그 안의 지역 변수들은
**클로저**에 갇혀서 게임이 끝날 때까지 살아 있습니다. 게임 루프는 이 변수들을
직접 읽고 씁니다. React는 이 사실을 모릅니다. **그리고 그게 정확히 우리가 원하는 것**입니다.

원칙:

> **초당 60번 변하는 것은 캔버스에 그린다. 가끔 변하는 것만 React에게 알린다.**

"가끔 변하는 것"을 알리는 유일한 통로가 92행입니다.

```ts
const pushUi = () => setUi({ phase, score, lives, best, eaten });
```

`pushUi()`는 쓰레기를 먹었을 때(299행), 맞았을 때(331행), 게임오버(312행), 시작(351행)
— 즉 **점수나 하트가 실제로 바뀌는 순간에만** 호출됩니다. 초당 60번이 아니라 초당 2~3번입니다.

📌 **학습 포인트**: 이 패턴은 게임뿐 아니라 지도, 차트, 에디터 등 "고빈도 렌더링 + 가끔 UI 갱신"이
필요한 모든 곳에 그대로 적용됩니다. 기억해 두세요.

## 3. 심장: 게임 루프 (777~784행)

```ts
const loop = (now: number) => {
  const dt = Math.min(0.05, (now - last) / 1000);  // ①
  last = now;
  t += dt;                                          // ②
  update(dt);                                       // ③
  draw();                                           // ④
  raf = requestAnimationFrame(loop);                // ⑤
};
```

게임은 **"시간을 조금 흘려보내고(update) → 그 순간을 사진 찍는다(draw)"**를 무한 반복하는 것뿐입니다.
이게 전부입니다. 진짜로.

### ① `dt` (델타 타임) — 게임 프로그래밍의 알파이자 오메가

`requestAnimationFrame`은 브라우저가 "지금 화면 그릴 준비 됐어"라고 부르는 콜백입니다.
문제는 **호출 간격이 일정하지 않다**는 것.

- 60Hz 모니터 → 약 16.7ms마다
- 120Hz 아이패드 → 약 8.3ms마다
- 렉 걸리면 → 40ms, 100ms…

만약 이렇게 짰다면:

```ts
// ❌ 프레임 의존적 코드
j.y += 2;   // "한 프레임에 2픽셀 내려간다"
```

120Hz 폰에서는 쓰레기가 **2배 빠르게** 떨어집니다. 게임이 사람마다 다른 게임이 됩니다.

그래서 "한 프레임에 몇 픽셀"이 아니라 **"1초에 몇 픽셀"**로 정의하고, 흐른 시간을 곱합니다.

```ts
// ✅ 실제 코드 (420행)
j.y += j.vy * dt;   // vy는 "초당 픽셀". dt는 "흐른 초".
```

`vy = 60`, `dt = 0.0167`(60Hz) → 프레임당 1px. `dt = 0.0083`(120Hz) → 프레임당 0.5px.
**결과적으로 두 기기에서 1초에 똑같이 60px 내려갑니다.** 이것이 `dt`의 존재 이유입니다.

### `Math.min(0.05, …)` 클램프는 왜?

브라우저 탭을 백그라운드로 보냈다가 30초 뒤 돌아오면 `dt`가 **30**이 됩니다.
그러면 420행에서 쓰레기가 한 프레임에 1800px 순간이동하며 플레이어를 통과해 버립니다
(이걸 **터널링(tunneling)**이라 부릅니다). 충돌 판정이 통째로 씹힙니다.

그래서 `dt`의 상한을 0.05초(=20FPS 상당)로 자릅니다. 20FPS보다 느리게 도는 프레임은
"슬로우 모션"이 될지언정 물리가 깨지지는 않습니다. **정확성보다 안정성을 택한 것.**

### ② `t` — 게임 시작 후 누적 시간(초)

이 변수 하나가 게임의 모든 "살아있음"을 담당합니다. `Math.sin(t * 속도)`는
-1 ↔ 1 을 왔다갔다하는 값이라, 뭐든 넣으면 흔들립니다.

| 코드 | 효과 |
|---|---|
| `Math.sin(t * 2) * 3` (654행) | 안테나가 초당 약 0.3회 좌우로 3px 흔들림 |
| `Math.sin(t * 5 + j.seed)` (605행) | 별이 뛰는 맥박. `+j.seed`로 별마다 위상이 달라 따로 논다 |
| `Math.floor(t * 16) % 2` (649행) | 초당 8번 켜졌다 꺼졌다 → 무적 깜빡임 |

📌 **`+ seed` 트릭**: 같은 `t`를 쓰면 모든 객체가 **동시에** 똑같이 움직여서 기계처럼 보입니다.
객체마다 다른 상수(`seed`)를 `sin` 안에 더해주면 위상이 어긋나서 자연스러워집니다.
게임/모션 그래픽에서 매우 자주 쓰는 기법입니다.

### ③④ update와 draw를 분리하는 이유

- `update`: 세상을 바꾼다. 캔버스를 절대 건드리지 않는다.
- `draw`: 세상을 읽기만 한다. 아무것도 바꾸지 않는다.

이 규칙을 지키면 "그리는 코드 때문에 게임 로직이 깨지는" 최악의 버그가 원천 봉쇄됩니다.
디버깅할 때도 "숫자가 이상하다 → update만 본다 / 화면이 이상하다 → draw만 본다"로 반이 잘립니다.

> ⚠️ 이 코드는 한 군데서 규칙을 어깁니다: `draw()`의 739행 `wobPh = …`와 743행 `Math.random()`.
> 순수한 렌더링 상태라 실용상 문제는 없지만, 원칙을 알고 어기는 것과 모르고 어기는 것은 다릅니다.

## 4. 좌표계와 DPR — 폰에서 선이 뿌옇지 않게 (806~829행)

캔버스에는 **두 개의 크기**가 있습니다. 이걸 헷갈리면 그림이 뭉개지거나 찌그러집니다.

| | 무엇 | 어디서 정함 |
|---|---|---|
| **CSS 크기** | 화면에서 차지하는 물리적 크기 | `className="absolute inset-0 h-full w-full"` (860행) |
| **버퍼 크기** | 실제 픽셀 격자의 해상도 | `canvas.width = w * dpr` (810행) |

아이폰은 `devicePixelRatio = 3`입니다. CSS 390px 폭이 실제로는 1170개의 하드웨어 픽셀입니다.
버퍼를 390으로 두면 390개 픽셀을 1170개로 늘려 그리므로 **선이 흐릿해집니다.**

```ts
const dpr = Math.min(window.devicePixelRatio || 1, 2);  // 807행
w = canvas.clientWidth;      // CSS 크기 (예: 390)
h = canvas.clientHeight;     // (예: 844)
canvas.width  = w * dpr;     // 버퍼 크기 (예: 780)
canvas.height = h * dpr;
ctx.setTransform(dpr, 0, 0, dpr, 0, 0);   // ← 핵심!
```

`setTransform(dpr, 0, 0, dpr, 0, 0)`은 "이제부터 내가 그리는 모든 좌표에 `dpr`을 곱해라"라는 뜻입니다.
덕분에 **게임 코드는 DPR을 완전히 잊고 CSS 좌표(0~390)로만 생각하면 됩니다.** 선명함은 공짜로 따라옵니다.

`Math.min(…, 2)`로 자르는 이유: DPR 3에서 버퍼는 픽셀 수가 9배(3²)가 됩니다. 채우는 비용도 9배.
2로 자르면 4배로 줄고, 육안 차이는 거의 없습니다. **화질 대비 성능의 실용적 타협점.**

## 5. 상태 모델

### 페이즈 머신 (63행)

```ts
let phase: Phase = "title";   // "title" | "playing" | "over"
```

상태 3개짜리 유한 상태 기계(FSM)입니다. 전이는 딱 두 개:

```
   ┌──────────────────────────── start() ───────────────────┐
   │                                                        │
title ──── 탭 ────▶ playing ──── lives===0 ────▶ over ──── 탭(0.6초 후) ─┘
```

`phase`가 게임 전체의 스위치입니다. 예를 들어:
- 426행 `if (phase !== "playing") continue;` → 타이틀/게임오버 화면에서는 충돌 판정 자체를 안 함
  (그래서 타이틀 화면 쓰레기는 캐릭터를 그냥 통과합니다. **의도된 동작**)
- 363~371행 → `playing`이면 손가락을 따라가고, 아니면 화면 중앙에서 둥둥 떠다님

📌 게임오버 직후 0.6초 입력 무시(799행 `if (t - overAt < 0.6) return;`)는
**죽은 순간 누르고 있던 손가락 때문에 게임오버 화면을 못 읽고 바로 재시작되는 것**을 막는 장치입니다.
이런 걸 **입력 디바운스**라 하고, 실무 게임에서 반드시 넣습니다.

### 엔티티는 그냥 "객체 배열"

```ts
let junks: Junk[]   = [];   // 떨어지는 쓰레기
let popups: Popup[] = [];   // "냠!" 같은 뜨는 글씨
let sparks: Spark[] = [];   // 먹을 때 튀는 입자
let bgStars: BgStar[] = []; // 배경 별
```

클래스도, 상속도, ECS도 없습니다. **평범한 객체가 든 평범한 배열 4개.**
소규모 게임에서는 이게 가장 빠르고 가장 읽기 쉽습니다.
(수천 개 단위로 커지면 그때 가서 구조를 바꾸면 됩니다. 미리 하지 마세요.)

## 6. 부드러운 움직임의 정체: 지수 감쇠 (363~366행)

캐릭터가 손가락을 **딱딱 따라가지 않고 살짝 늦게 미끄러지듯** 따라오는 게
이 게임 손맛의 8할입니다. 그 코드는 이게 전부입니다.

```ts
const k = Math.min(1, dt * 7);
player.x += (player.tx - player.x) * k;   // tx = 손가락 목표 위치
player.y += (player.ty - player.y) * k;
```

읽는 법: **"목표까지 남은 거리의 일부(k)만큼 매 프레임 다가간다."**

```
프레임 1: 거리 100 → 12 이동 → 남은 거리 88
프레임 2: 거리  88 → 10 이동 → 남은 거리 78
프레임 3: 거리  78 →  9 이동 → 남은 거리 69
...
남은 거리가 줄면 이동량도 줄어든다 → 목표 근처에서 자연스럽게 감속 (ease-out)
```

`* 7`이 **따라오는 속도(반응성)**입니다. 이 숫자 하나만 바꿔보면 감이 옵니다:

| 값 | 느낌 |
|---|---|
| `dt * 2` | 굼벵이. 젤리 속에서 헤엄치는 느낌 |
| `dt * 7` | 현재 값. 쫀득하게 따라옴 |
| `dt * 30` | 거의 즉시. 손가락에 딱 붙음 |
| `k = 1` | 순간이동. 부드러움 0 |

`Math.min(1, …)`이 필요한 이유: 만약 `dt`가 크면(렉) `dt * 7 > 1`이 되어
`x += 거리 * 1.5` → **목표를 지나쳐 버리고 진동**합니다. 1로 잘라서 오버슈트를 막습니다.

이 패턴은 코드 곳곳에서 재사용됩니다 — 입 벌리기(392행), 눈동자 추적(397행), 쓰레기 빨려들어가기(409행).
**한 줄짜리 관용구(idiom)로 외워두면 평생 씁니다.**

<details>
<summary>🎓 심화: 사실 이건 프레임 독립적이지 않습니다</summary>

엄밀히 말해 `x += (target - x) * dt * k`는 `dt`가 달라지면 **결과가 미묘하게 다릅니다.**
(60Hz에서 두 프레임 진행 ≠ 30Hz에서 한 프레임 진행)

수학적으로 정확한 프레임 독립 버전은:

```ts
const k = 1 - Math.exp(-7 * dt);   // 지수 감쇠의 해석해
player.x += (player.tx - player.x) * k;
```

`dt`가 작을 때 `1 - exp(-7dt) ≈ 7dt`라서 현재 코드는 **좋은 근사**이고,
`dt`가 0.05로 클램프되어 있어 오차가 눈에 띄지 않습니다. 그래서 대부분의 게임이 그냥 곱셈을 씁니다.
하지만 "왜 근사인지" 알고 쓰는 것과 모르고 쓰는 것은 다릅니다.
정밀한 물리(리플레이, 네트워크 동기화)가 필요하면 `exp` 버전을 쓰세요.
</details>

## 7. 스폰과 난이도 곡선 (247~272, 356~361행)

### 스폰 타이머

```ts
spawnIn -= dt;
if (spawnIn <= 0) {
  spawn();
  const base = phase === "playing" ? Math.max(0.42, 1.05 - score / 900) : 1.5;
  spawnIn = base * (0.7 + Math.random() * 0.6);   // ±30% 랜덤
}
```

- **카운트다운 방식**: "몇 초 후에 다음 놈"을 담아두고 `dt`만큼 깎습니다. `setInterval`을 쓰지 않는 이유는
  게임 시간(일시정지/느린 프레임)과 어긋나면 안 되기 때문입니다.
- **`1.05 - score / 900`**: 점수 0에서 1.05초 간격 → 점수 900에서 0.05초… 가 아니라
  `Math.max(0.42, …)`로 바닥이 0.42초입니다. **난이도가 무한히 어려워지지 않도록 천장(floor)을 둔 것.**
  이걸 안 두면 후반에 화면이 쓰레기로 도배됩니다.
- **`* (0.7 + random * 0.6)`**: 간격을 ±30% 흔듭니다. 정확히 일정한 리듬은
  플레이어가 패턴을 외워버려서 지루해집니다. **불규칙성은 의도적으로 설계하는 것.**

### 무엇이 나오는가 (248~253행)

## 8. 충돌 판정 — 그냥 거리 재기 (428~442행)

물리 엔진이 필요할 것 같지만, **원과 원의 충돌**은 피타고라스 정리 한 줄입니다.

```ts
const d = Math.hypot(j.x - player.x, j.y - player.y);   // 두 중심 사이 거리
// Math.hypot(a, b) === Math.sqrt(a*a + b*b)
```

```
    (player.x, player.y)          (j.x, j.y)
            ●─────────d─────────────●
           ╱ ╲                     ╱ ╲
      player.r                  j.size

  d < player.r + j.size  →  두 원이 겹쳤다 = 충돌!
```

### 히트박스는 눈에 보이는 것과 다르다

**이게 실무의 핵심 노하우입니다.** 코드는 겉보기 크기를 그대로 쓰지 않습니다.

```ts
// 위험물 (430행) — 히트박스를 작게
if (player.invul <= 0 && d < player.r + j.size * 0.75)

// 먹을 것 (441행) — 히트박스를 크게
if (d < player.r + j.size * 0.65 + player.mouth * 10)
```

- 위험물의 **그림**은 반지름 `size * 1.5`짜리 뾰족뾰족(621행)인데 **판정**은 `size * 0.75`.
  → 가시 끝이 스쳐도 안 죽습니다. "어? 안 맞았는데 죽었어!"라는 억울함을 없앱니다.
- 먹을 것은 **입을 벌린 정도(`player.mouth`, 0~1)만큼 판정이 커집니다(`+ mouth * 10`).**
  → 입 크게 벌리면 실제로 더 잘 먹힙니다. 연출과 판정이 일치해서 직관적입니다.

> 🎮 **황금률: 플레이어에게 유리한 쪽으로 거짓말하라.**
> 피격 판정은 보이는 것보다 작게, 획득 판정은 보이는 것보다 크게.
> 상용 게임(마리오, 젤다, 슈팅게임 전부)이 이렇게 합니다.

### 자석 (436~440행) — 모바일이라서 필요한 것

```ts

  const pull = Math.min(1, dt * 3);
  j.x0 += (player.x - j.x) * pull;  // ← x가 아니라 x0!!
  j.y  += (player.y - j.y) * pull;

```

작은 폰 화면에서 손가락으로 픽셀 단위 조준은 불가능합니다. 그래서 **근처에 오면 살짝 끌어당깁니다.**
플레이어는 "내가 잘 먹었다"고 느끼지만 사실 게임이 도와준 것입니다.

⚠️ **여기서 반드시 짚고 갈 함정**: `j.x`가 아니라 **`j.x0`**를 수정합니다. 왜?

419행을 보세요.
```ts
j.x = j.x0 + Math.sin(j.swayT) * j.swayAmp;   // 매 프레임 x를 x0로부터 새로 계산!
```

쓰레기의 `x`는 **매 프레임 `x0`(좌우 흔들림의 중심축)으로부터 다시 계산**됩니다.
그러니 `j.x`를 아무리 고쳐봤자 **다음 프레임에 즉시 덮어써집니다.** 아무 효과가 없습니다.
진짜로 옮기려면 **중심축인 `x0`를 옮겨야** 합니다.

📌 **일반 원칙: 파생된 값(derived value)을 수정하지 말고, 원본을 수정하라.**
이 버그는 초보자가 100% 밟는 지뢰입니다. 기억해 두세요.
(반면 `y`는 파생값이 아니라 누적값(`y += vy * dt`)이라 직접 수정해도 됩니다. 그래서 `j.y`는 그냥 씁니다.)

## 10. ⭐ 이 코드의 하이라이트: 손그림 렌더링

여기가 이 게임의 정체성입니다. 캔버스에 삐뚤빼뚤한 선을 그리는 원리를 뜯어봅니다.

### 10-1. `wob()` — 결정론적 가짜 난수 (139~142행)

```ts
const wob = (seed: number, i: number, ph: number) => {
  const v = Math.sin(seed * 127.1 + i * 311.7 + ph * 74.7) * 43758.5453;
  return (v - Math.floor(v)) * 2 - 1;   // 결과: -1 ~ +1
};
```

**동작 원리를 한 줄씩:**
1. `sin(...)` → -1 ~ 1 사이의 매끄러운 값
2. `* 43758.5453` → 엄청나게 큰 수로 뻥튀기. 입력이 조금만 달라져도 결과가 확 달라짐
3. `v - Math.floor(v)` → **소수 부분만 남김** (예: 12345.678 → 0.678). 이걸 `fract`라고 부릅니다.
   큰 수의 소수 부분은 사실상 예측 불가 → **난수처럼 보임**
4. `* 2 - 1` → 0~1 범위를 -1~1 로 변환

이 이상한 마법 상수들(127.1, 311.7, 43758.5453)은 **셰이더 프로그래밍 세계의 유명한 관용구**입니다.
GLSL의 `fract(sin(dot(uv, vec2(12.9898,78.233))) * 43758.5453)` 를 본 적 있다면 같은 물건입니다.

### 🔑 왜 `Math.random()`을 안 쓰고 이 짓을 하는가?

**이것이 가장 중요한 질문입니다.**

`Math.random()`은 부를 때마다 다른 값을 줍니다. 그러면 매 프레임 캐릭터의 삐뚤빼뚤함이
완전히 새로 뽑혀서, **캐릭터가 부들부들 떨며 형체가 유지되지 않습니다.** (한번 해보세요. 끔찍합니다.)

`wob(seed, i, ph)`는 **같은 입력 → 항상 같은 출력**입니다. (순수 함수)
그래서 `seed`(어느 물체)와 `i`(그 물체의 몇 번째 점)가 같으면
`ph`를 바꾸지 않는 한 **매 프레임 똑같은 삐뚤빼뚤함**이 나옵니다. 형체가 유지됩니다.

### 10-2. `wobPh` — "부글부글 끓는 선"의 비밀 (739행)

```ts
wobPh = Math.floor(t * 7);   // draw() 첫 줄
```

`t`는 매끄럽게 증가하지만, `Math.floor(t * 7)`은 **초당 7번만 계단식으로 바뀌는 정수**입니다.

```
t (연속):     0.00  0.05  0.10  0.15  0.20  0.25  0.30  0.35
wobPh:          0     0     0     1     1     1     2     2
              └─ 같은 모양 ─┘ └─ 다른 모양 ─┘ └─ 또 다른 모양 ─┘
```

- 6프레임 정도는 **완전히 똑같은 삐뚤빼뚤함**으로 그려짐 (형체 유지)
- 그러다 `wobPh`가 바뀌는 순간 **모든 선이 한꺼번에 새 모양으로 튐**

이게 바로 손으로 그린 애니메이션 특유의 **"boiling line(끓는 선)"** 효과입니다.
연필로 같은 그림을 여러 장 그려서 넘기면 선이 미묘하게 다르죠? 그걸 흉내낸 겁니다.
초당 7번은 대략 **"1초에 7장짜리 애니메이션"**을 그리는 셈입니다.

**실험해 보세요** (제일 재밌는 실습입니다):

| 코드 | 결과 |
|---|---|
| `wobPh = 0` | 삐뚤빼뚤하지만 **정지**. 종이에 그린 정물화 |
| `wobPh = Math.floor(t * 3)` | 느긋하게 꿈틀. 나른한 느낌 |
| `wobPh = Math.floor(t * 7)` | **현재 값.** 활기찬 낙서 애니메이션 |
| `wobPh = Math.floor(t * 30)` | 지지직. 눈이 아픔 |
| `wobPh = t` (floor 없이) | 매끄럽게 일렁임. **손그림이 아니라 물속 해파리처럼 보임** |

마지막 줄이 핵심입니다. **`Math.floor`(양자화)가 없으면 "손그림"이 아닙니다.**
디지털적인 딱딱 끊김이 오히려 아날로그 느낌을 만드는 역설.

### 10-3. `wobblyPath()` — 직선을 흔들기 (144~167행)

```ts
const wobblyPath = (pts: Pt[], seed: number, amp: number, close: boolean) => {
```

받은 꼭짓점들 사이를 **각 변마다 2등분**해서 점을 늘리고, 그 점들을 좌우로 `amp`만큼 흔듭니다.

```
입력: 사각형 꼭짓점 4개
   A ────────────── B          A ─·─ ~ ─·─ B      (·  = 추가된 중간점)
   │                │    →     ╎           ╎       (~  = 흔들린 선)
   │                │          ╎           ╎
   D ────────────── C          D ─·─ ~ ─·─ C

점을 안 늘리면: 꼭짓점만 흔들려서 여전히 "직선"으로 보임 (변이 곧음)
점을 늘리면:    변 중간도 휘어서 손으로 그은 선처럼 보임
```

`for (let s = 0; s < 2; s++)`가 그 세분화(subdivision)입니다.
`s`를 3, 4로 늘리면 더 구불구불해집니다. (`const SUB = 2`로 뽑아서 실험해 보세요.)

`close` 파라미터: `true`면 닫힌 도형(사각형, 별), `false`면 열린 선(스프링, 안테나).
`pts[(i + 1) % n]`의 `% n`이 **마지막 점 → 첫 점**을 이어주는 부분입니다.

### 10-4. `wobblyBlob()` — 삐뚤빼뚤한 원 (169~186행)

캐릭터 몸통, 달, 입에 쓰이는 함수. 두 단계로 나뉩니다.

**1단계: 원 위에 점 12개를 찍되, 반지름을 흔든다**
```ts
for (let i = 0; i < 12; i++) {
  const a = (i / 12) * Math.PI * 2;        // 0° ~ 360° 를 12등분
  const rr = r + wob(seed, i, wobPh) * amp; // ← 반지름이 점마다 다름!
  pts.push([x + Math.cos(a) * rr, y + Math.sin(a) * rr]);
}
```
`cos`는 x, `sin`은 y — **각도를 좌표로 바꾸는 공식**입니다. 원을 그리는 기본 중의 기본.
반지름 `rr`을 흔들었으니 완벽한 원이 아니라 감자 같은 모양이 됩니다.

**2단계: 그 점들을 부드러운 곡선으로 잇는다 — "중점 기법"**
```ts
let m = mid(pts[11], pts[0]);
ctx.moveTo(m[0], m[1]);
for (let i = 0; i < 12; i++) {
  m = mid(pts[i], pts[(i + 1) % 12]);
  ctx.quadraticCurveTo(pts[i][0], pts[i][1], m[0], m[1]);
  //                   └ 제어점 = 꼭짓점 ┘  └ 끝점 = 다음 중점 ┘
}
```

이게 캔버스 곡선 그리기의 **정석 관용구**입니다. 반드시 이해하고 넘어가세요.

```
   P0 ─────── P1 ─────── P2        Pi = 원래 점 (꼭짓점)
       ╲  M0  ╱   ╲  M1 ╱          Mi = Pi와 P(i+1)의 중점
        ╲___╱      ╲__╱

  quadraticCurveTo(제어점=P1, 끝점=M1) 를 M0에서 시작해 호출
  → 곡선이 M0에서 출발해 P1 쪽으로 당겨졌다가 M1에 도착
```

**왜 점이 아니라 중점을 지나게 할까?**
`quadraticCurveTo`는 제어점을 **지나가지 않고 스쳐 지나갑니다**(당겨질 뿐).
만약 끝점을 `P2`로 잡으면 P2에서 곡선이 **뾰족하게 꺾입니다**(미분 불연속).
끝점을 중점으로 잡으면 이전 곡선의 끝 방향과 다음 곡선의 시작 방향이 자동으로 일치해서
**전체가 매끄럽게(C1 연속) 이어집니다.** 수학적 보장입니다.

📌 이 기법은 손글씨 앱, 서명 패드, 그림판 등 **모든 부드러운 자유곡선**에 쓰입니다.
`lineTo`로 이으면 각져 보이고, `bezierCurveTo`는 제어점 계산이 복잡합니다. 중점 기법이 최적점입니다.

### 10-5. `fillStroke()` — 색연필 채색 (211~219행)

```ts
const fillStroke = (color: string, fillAlpha = 0.15) => {
  const ga = ctx.globalAlpha;
  ctx.globalAlpha = ga * fillAlpha;   // ① 곱한다 (덮어쓰지 않는다!)
  ctx.fillStyle = color;
  ctx.fill();                          // 연하게 속을 칠하고
  ctx.globalAlpha = ga;                // ② 원복
  ctx.strokeStyle = color;
  ctx.stroke();                        // 같은 색으로 진하게 외곽선
};
```

**같은 색으로 "속은 연하게 + 선은 진하게"** 칠하면 색연필/마커로 칠한 느낌이 납니다.
채도 높은 단색으로 꽉 채우면 벡터 아이콘처럼 딱딱해집니다.

⚠️ **`ga * fillAlpha`로 곱하는 게 핵심입니다.** `ctx.globalAlpha = fillAlpha`로 덮어쓰면
무적 깜빡임(649행에서 `globalAlpha = 0.4`로 설정) 중에 몸통만 깜빡이지 않는 버그가 납니다.
**바깥에서 걸어둔 투명도를 존중하려면 곱셈, 무시하려면 대입.** 항상 곱셈이 안전합니다.

> 💡 **개발 중 실제로 났던 버그**: 처음엔 `fillAlpha` 기본값 0.15로 캐릭터 몸통을 칠했는데,
> 어두운 배경(#141838) 위에서 몸 속이 거의 안 보여서 **투명한 유령**처럼 보였습니다.
> 667행에서 `0.34`로 올려서 해결했습니다. **배경색과 채움 투명도는 항상 같이 튜닝해야 합니다.**

## 11. 캔버스 그리기 실전 (527~736행)

### `save()` / `restore()` — 절대 빼먹으면 안 되는 것

```ts
ctx.save();                  // 현재 좌표계·스타일을 스택에 백업
ctx.translate(j.x, j.y);     // 원점을 물체 위치로 이동
ctx.rotate(j.rot);           // 원점 기준으로 회전
ctx.scale(sc, sc);           // 원점 기준으로 확대/축소
// ...  이제 (0, 0) 기준으로 그리면 됨! 물체의 화면 좌표를 몰라도 된다
ctx.restore();               // 백업 복원 (안 하면 다음 물체까지 회전됨!!)
```

**`translate` → `rotate` → 로컬 좌표로 그리기**는 캔버스의 핵심 패턴입니다.
이걸 안 쓰면 회전한 사각형의 네 꼭짓점을 삼각함수로 직접 계산해야 합니다. 지옥입니다.

> ⚠️ `save()` 없이 `rotate()`하면 **그 이후 모든 그리기가 누적 회전**됩니다.
> 화면이 빙글빙글 돌기 시작하면 십중팔구 `restore()`를 빼먹은 것입니다.
> **`save`와 `restore`는 괄호처럼 반드시 짝을 맞추세요.**

### 위성 하나 뜯어보기 (538~563행)

```ts
for (const dir of [-1, 1] as const) {     // 좌우 태양전지판 (대칭이니 루프로)
  const px = dir === 1 ? s * 0.85 : -s * 1.95;
  wobblyPath(rectPts(px, -s * 0.38, s * 1.1, s * 0.76), j.seed + dir, 1.5, true);
  fillStroke("#ffd166", 0.13);            // 노란 패널
  wobblyLine(…);  strokeOnly("#ffd166");  // 패널 가운데 분할선
}
wobblyPath(rectPts(-s*0.75, -s*0.55, s*1.5, s*1.1), j.seed + 7, 1.6, true);
fillStroke(c);                            // 본체
// … 안테나, 눈 2개, 웃는 입
```

배울 점 두 가지:

1. **모든 치수가 `s`(=`j.size`)의 배수입니다.** 하드코딩된 픽셀이 없습니다.
   덕분에 `size`만 바꾸면 모든 비율이 유지된 채로 커지고 작아집니다. **비례 기반 디자인.**
2. **`j.seed + 1`, `j.seed + 7`처럼 시드를 어긋나게 줍니다.** 같은 시드를 주면
   왼쪽 패널과 오른쪽 패널이 **똑같은 방향으로 삐뚤어져서** 부자연스럽습니다.
   시드를 다르게 하면 각 부품이 독립적으로 삐뚤어집니다.

### 눈, 입, 볼 — "귀여움"의 공식 (669~734행)

캐릭터가 귀여운 건 우연이 아니라 다음 4개 장치의 합입니다. 하나씩 지워보면 압니다.

| 장치 | 줄 | 원리 |
|---|---|---|
| **눈동자 추적** | 691행 | `look` 벡터(정규화된 방향)만큼 눈동자를 눈 안에서 이동. 가장 가까운 먹이를 쳐다봄 |
| **눈 깜빡임** | 674~680행 | `blinkT > 0`이면 원 대신 아래로 볼록한 **호(arc)** 하나. 0.13초. 2.2~4.7초마다 |
| **볼터치** | 697~706행 | 분홍 타원 2개 `globalAlpha 0.4`. 이거 하나로 체온이 생깁니다 |
| **입 벌리기** | 711~727행 | `mouth`(0~1)에 따라 선(∪) → 검은 구멍 + 분홍 혀 |

눈동자 추적의 수학 (393~398행):
```ts
const dx = nearestJunk.x - player.x;
const dy = nearestJunk.y - player.y;
const dd = Math.hypot(dx, dy) || 1;      // 거리 (0 나눗셈 방어용 || 1)
player.look.x += (dx / dd - player.look.x) * Math.min(1, dt * 6);
//                └ 정규화: 거리로 나누면 길이 1짜리 "방향만" 남음 ┘
//                                              └ 또 지수 감쇠! 눈동자도 부드럽게 ┘
```
**벡터 정규화**(길이를 1로 만들기)는 "거리는 버리고 방향만 쓰고 싶을 때" 씁니다.
그래야 먹이가 멀든 가깝든 눈동자가 눈 밖으로 튀어나가지 않습니다.

입 벌리기의 세로 늘이기 트릭 (714~718행):
```ts
ctx.save();
ctx.translate(x, my);
ctx.scale(1, mry / mrx);        // ← y축만 늘림
wobblyBlob(0, 0, mrx, 3.3, 1.4); // 원(blob)을 그리면
ctx.restore();                   // 화면에는 타원으로 나옴!
```
`wobblyBlob`은 **원**만 그릴 수 있습니다. 타원이 필요하면 함수를 고치는 대신
**좌표계를 찌그러뜨린 상태에서 원을 그립니다.** 함수 하나를 여러 용도로 재활용하는 좋은 방법입니다.

## 12. 파티클과 팝업 — "수명 기반" 설계

### 공통 패턴

```ts
type Spark = { …, age: number, life: number };   // 나이 / 수명
```

두 필드로 모든 걸 합니다:

```ts
s.age += dt;                                    // 나이를 먹고
if (s.age >= s.life) sparks.splice(i, 1);       // 수명이 다하면 제거
ctx.globalAlpha = Math.max(0, 1 - s.age / s.life);  // 나이 비율만큼 흐려짐
//                              └ 0 → 1 로 가는 진행도 ┘
```

`age / life`는 **0에서 1로 가는 정규화된 진행도**입니다. 이걸 알파, 크기, 위치 등
아무 데나 곱하면 애니메이션이 됩니다. 게임 이펙트의 만능 공식입니다.

### 팝업의 지연 페이드 (763행)

```ts
const f = p.age / p.life;                       // 0 → 1
ctx.globalAlpha = f < 0.7 ? 1 : 1 - (f - 0.7) / 0.3;
```

처음 70%는 **완전 불투명**, 마지막 30%에서만 사라집니다.
처음부터 서서히 흐려지면 **글자를 읽기 전에 이미 반투명**이라 잘 안 보입니다.
**"충분히 보여준 다음 사라진다"** — 텍스트 피드백의 기본기입니다.

### 팝업 외곽선 (767~771행)

```ts
ctx.lineWidth = 5;
ctx.strokeStyle = BG;          // 배경색으로 두껍게 외곽선을 먼저 긋고
ctx.strokeText(p.text, 0, 0);
ctx.fillStyle = p.color;
ctx.fillText(p.text, 0, 0);    // 그 위에 글자를 채움
```

배경색으로 글자 테두리를 두껍게 그리면 **어떤 배경 위에서도 글자가 읽힙니다.**
지도 라벨, 자막, 게임 데미지 숫자가 전부 이 기법을 씁니다. (CSS의 `text-stroke`와 같은 원리)

### 파티클 터뜨리기 (230~245행)

```ts
const a = Math.random() * Math.PI * 2;   // 0~360° 무작위 방향
const sp = 40 + Math.random() * 120;     // 무작위 속력
vx: Math.cos(a) * sp,                    // 방향+속력 → 속도 벡터
vy: Math.sin(a) * sp,
```
**"방향(각도) + 속력(크기) → 속도 벡터"**. 폭발/분출 이펙트의 표준 공식입니다.
중력을 넣고 싶으면 update에 `s.vy += 400 * dt;` 한 줄만 추가하면 됩니다. (실습 과제에 있습니다)

### 화면 흔들기 (741~744행)

```ts
if (shake > 0) {
  const a = (shake / 0.35) * 7;   // 남은 시간 비율 × 최대 7px
  ctx.translate((Math.random() - 0.5) * a, (Math.random() - 0.5) * a);

```
**전체 좌표계를 매 프레임 무작위로 살짝 밀어버립니다.** 개별 물체를 흔드는 게 아닙니다.

`drawBackground`가 `-12, -12, w+24, h+24`로 **화면보다 크게** 배경을 칠하는 이유(464행)가 이겁니다.
안 그러면 흔들릴 때 **가장자리에 빈 틈**이 보입니다.

## 13. 사운드 — 오디오 파일 0개짜리 신시사이저 (94~135행)

mp3 파일 하나 없이 **WebAudio로 소리를 직접 합성**합니다.

```ts
const blip = (f0, f1, dur, type = "triangle", vol = 0.1, delay = 0) => {
  const o = ac.createOscillator();   // 발진기: 순수한 파형을 만드는 것
  const g = ac.createGain();         // 볼륨 조절기
  o.type = type;                     // "sine"(부드러움) "triangle"(말랑) "sawtooth"(거침)
  o.frequency.setValueAtTime(f0, t0);
  o.frequency.exponentialRampToValueAtTime(Math.max(30, f1), t0 + dur);  // 음정 미끄러짐
  g.gain.setValueAtTime(vol, t0);
  g.gain.exponentialRampToValueAtTime(0.0001, t0 + dur);   // 볼륨 페이드아웃
  o.connect(g); g.connect(ac.destination);   // 발진기 → 볼륨 → 스피커
  o.start(t0); o.stop(t0 + dur + 0.05);
};
```

**소리 디자인의 문법**을 그대로 보여줍니다:

| 효과음 | 코드 | 왜 그렇게 들리는가 |
|---|---|---|
| 먹기 | `blip(430 → 900, 0.09초, triangle)` | 음정이 **올라감** = 긍정/획득 |
| 피격 | `blip(220 → 65, 0.28초, sawtooth)` | 음정이 **떨어짐** + 거친 톱니파 = 부정/실패 |
| 별 | 660 → 880 → 1320, 0.08초씩 지연 | **아르페지오**(도-미-솔 분산화음) = 화려한 보상 |
| 게임오버 | 392 → 330 → 262 → 130 | **하강 음계** = 낙담 |

> 🎵 **규칙: 올라가는 음 = 좋은 일, 내려가는 음 = 나쁜 일.** 마리오 코인 소리와 죽는 소리를 떠올려 보세요.
> 인간의 보편적 감각이라 문화권을 안 탑니다.

`exponentialRampToValueAtTime`을 쓰는 이유: 인간의 청각은 **로그 스케일**입니다.
음량/음정을 선형으로 줄이면 "뚝 끊기는" 느낌이 나고, 지수적으로 줄여야 자연스럽습니다.
(그래서 목표값이 0이 아니라 `0.0001`입니다. 지수 함수는 0에 도달할 수 없어서 0을 넣으면 에러납니다!)

### ⚠️ 자동재생 정책 — 반드시 알아야 할 브라우저 규칙

```ts
const ensureAudio = () => {
  if (!ac) ac = new AudioContext();
  if (ac.state === "suspended") void ac.resume();   // ← 반드시 사용자 제스처 안에서!
};
// 794~796행
const onDown = (e: PointerEvent) => {
  e.preventDefault();
  ensureAudio();   // 첫 탭에서 오디오 활성화
  …
};
```

**모든 모던 브라우저는 사용자가 클릭/탭하기 전에는 소리를 못 내게 막습니다.**
(광고가 갑자기 소리 지르는 걸 막으려고 도입됨)

`AudioContext`를 페이지 로드 시점에 만들면 `state === "suspended"` 상태로 태어나고,
**사용자 제스처 이벤트 핸들러 안에서 `resume()`을 불러야만** 깨어납니다.
그래서 `ensureAudio()`가 `pointerdown` 핸들러에 있는 것입니다. 게임 시작 탭이 곧 오디오 활성화 탭입니다.

103행 `if (!ac || ac.state !== "running") return;` — 아직 안 깨어났으면 조용히 아무것도 안 합니다.
소리 안 난다고 게임이 죽으면 안 되니까요. **부가 기능은 항상 실패해도 되게 만드세요.**

## 14. 모바일 대응 총정리

이 게임이 폰에서 잘 되는 건 다음 7가지 장치 덕분입니다. **하나라도 빠지면 폰에서 망가집니다.**

### ① 뷰포트 메타 (`app/play/page.tsx`)

```ts
export const viewport: Viewport = {
  width: "device-width",
  initialScale: 1,
  maximumScale: 1,      // 확대 금지
  userScalable: false,  // 핀치 줌 금지 ← 게임 중 두 손가락으로 화면이 확대되면 재앙
  viewportFit: "cover", // 노치 영역까지 배경 채우기
  themeColor: "#141838",// 안드로이드 상단 바까지 배경색으로 → 몰입감
};
```
> Next.js 16에서는 `metadata`가 아니라 **별도의 `viewport` export**로 분리되었습니다.
> `metadata` 안에 `viewport`를 넣으면 무시됩니다. (구버전 튜토리얼 주의)

### ② `touch-action: none` (860행의 `touch-none`) — **제일 중요**

```html
<canvas className="… touch-none" />
```
이게 없으면 손가락을 드래그할 때 **브라우저가 "스크롤하려는 거구나"라고 판단해서
`pointermove` 이벤트를 끊어버립니다** (포인터 캡처 취소). 캐릭터가 중간에 멈춥니다.
`touch-action: none`은 "이 요소에서는 브라우저 기본 터치 제스처 다 꺼"라는 선언입니다.

### ③ 포인터 이벤트로 통일 (834~835행)

```ts
canvas.addEventListener("pointerdown", onDown);
canvas.addEventListener("pointermove", onMove);
```
`mousedown`+`touchstart`를 각각 다는 대신 `pointerdown` 하나로 **마우스·터치·펜을 모두** 처리합니다.
데스크톱에서 테스트하고 폰에서 그대로 굴러가는 이유입니다. 코드가 절반이 됩니다.

### ④ 손가락 오프셋 72px (790행) — **모바일 UX의 정수**

```ts
const py = e.clientY - rect.top - (e.pointerType === "touch" ? 72 : 0);
//                                 └ 터치일 때만 72px 위로 ┘
```
**손가락이 캐릭터를 가립니다.** 정확히 손가락 위치에 캐릭터를 두면 아무것도 안 보입니다.
그래서 터치일 때만 캐릭터를 손가락보다 72px 위에 띄웁니다.
마우스(`pointerType === "mouse"`)일 때는 가릴 게 없으니 오프셋 0.

**같은 코드가 입력 장치에 따라 다르게 동작해야 하는 대표 사례**입니다.

### ⑤ 경계 클램프 (791~792행)

```ts
player.tx = Math.min(Math.max(px, player.r + 6), w - player.r - 6);
player.ty = Math.min(Math.max(py, player.r + 64), h - player.r - 10);
//                                          └ 64: HUD(점수/하트) 아래로 못 올라가게 ┘
```
`Math.min(Math.max(값, 하한), 상한)` = **클램프 관용구**. 캐릭터가 화면 밖으로 못 나갑니다.
위쪽만 64px인 이유는 HUD와 겹치지 않게 하려는 것입니다.

### ⑥ Safe Area (865행)

```ts
style={{ paddingTop: "calc(env(safe-area-inset-top) + 10px)" }}
```
아이폰 노치/다이내믹 아일랜드 **아래로** HUD를 밀어냅니다. `env(safe-area-inset-*)`는
브라우저가 기기별로 채워주는 CSS 변수입니다. (`viewportFit: "cover"`와 세트로 써야 작동합니다)

### ⑦ 실수 방지 (849~851행)

```tsx
className="fixed inset-0 select-none overflow-hidden"   // 스크롤 봉쇄, 텍스트 선택 방지
style={{ WebkitTouchCallout: "none" }}                  // iOS 롱프레스 메뉴 방지
onContextMenu={(e) => e.preventDefault()}               // 우클릭/롱프레스 메뉴 방지
```
게임 중 실수로 텍스트가 파랗게 선택되거나, 롱프레스로 "이미지 저장하기" 팝업이 뜨면 몰입이 깨집니다.

## 15. 폰트: 한글 손글씨를 쓰는 법 (853~858행)

```tsx
<link rel="preconnect" href="https://fonts.gstatic.com" crossOrigin="anonymous" />
<link
  rel="stylesheet"
  precedence="default"                                  // ← React 19 기능!
  href="https://fonts.googleapis.com/css2?family=Gaegu:wght@400;700&display=swap"
/>
```

**왜 `next/font`를 안 썼나?**
`next/font/google`은 폰트를 빌드 타임에 다운로드해 셀프 호스팅해줘서 원래는 최선입니다. 그런데:

```bash
$ node_modules/next/.../font-data.json 확인 결과
"Gaegu": { "weights": ["300","400","700"], "subsets": ["latin"] }
                                                       ^^^^^^^ 한글(korean) 서브셋이 없음!
```
`next/font`는 `subsets`를 명시해야 하는데 Gaegu에는 `korean` 서브셋이 등록되어 있지 않습니다.
그래서 **Google Fonts CSS를 직접 링크**했습니다. (`display=swap` = 폰트 로딩 중엔 대체 폰트로 먼저 보여줌)

**`precedence="default"`는 React 19의 새 기능입니다.** 이게 붙은 `<link rel="stylesheet">`는
컴포넌트 어디에 쓰여 있든 React가 **자동으로 `<head>`로 끌어올리고(hoisting), 중복을 제거**합니다.
예전처럼 `next/head`나 `_document`를 건드릴 필요가 없습니다.

**캔버스에서 폰트 쓰기** (518, 766행):
```ts
ctx.font = `700 ${p.size}px ${DOODLE_FONT}`;   // "굵기 크기px 이름" — CSS font 단축 문법
ctx.textAlign = "center";     // 좌표를 글자 중앙에 맞춤
ctx.textBaseline = "middle";  // (기본값은 왼쪽/알파벳베이스라인이라 중앙 정렬하려면 필수)
```
⚠️ 캔버스는 폰트가 아직 로딩 안 됐으면 **말없이 대체 폰트로 그립니다.** 에러도 안 납니다.
그래서 첫 1초 정도는 글자가 다른 폰트로 보일 수 있습니다. `DOODLE_FONT`에
`"Comic Sans MS", "Chalkboard SE", cursive` 대체 폰트를 나열해 둔 이유입니다.

## 16. 정리(cleanup)와 메모리 누수 (838~844행)

```ts
return () => {
  cancelAnimationFrame(raf);                          // ① 루프 정지
  window.removeEventListener("resize", resize);       // ② 리스너 해제
  canvas.removeEventListener("pointerdown", onDown);
  canvas.removeEventListener("pointermove", onMove);
  if (ac) void ac.close().catch(() => {});            // ③ 오디오 컨텍스트 해제
};
```

**이 함수가 없으면 무슨 일이 벌어지는가:**

`/play`에서 `/`로 이동하면 React가 컴포넌트를 언마운트합니다. 하지만
`requestAnimationFrame` 루프는 **자기가 자기를 다시 예약**하므로(783행) 계속 돕니다.
사라진 캔버스에 계속 그림을 그리며 CPU와 배터리를 먹습니다. 페이지를 몇 번 왔다갔다하면
**루프가 여러 개 겹쳐 돕니다.** 이것이 전형적인 메모리/CPU 누수입니다.

### 🔥 Next.js는 개발 모드에서 StrictMode를 켭니다

React StrictMode는 **개발 환경에서 `useEffect`를 일부러 두 번 실행**합니다
(마운트 → 언마운트 → 재마운트). **정리 함수가 제대로 안 짜여 있으면 즉시 티가 나도록** 하는 장치입니다.

정리 함수가 없다면 개발 모드에서 게임 루프가 **2개** 돌면서 모든 게 2배속으로 움직입니다.
"개발할 땐 빠른데 배포하면 정상이에요" 같은 괴현상이 여기서 나옵니다.

> **규칙: `useEffect`에서 뭔가를 시작했으면(타이머, 루프, 리스너, 소켓, 옵저버)
> 정리 함수에서 반드시 멈춰라.** 예외 없습니다.

## 19. 이 게임에서 배운 것 총정리

한 문장으로 요약할 수 있는 것들만 모았습니다. **이것만 기억해도 절반은 가져간 것입니다.**

**게임 루프**
- 게임 = `update(dt)` + `draw()`의 무한 반복. 그 이상도 이하도 아니다.
- 모든 속도는 "초당"으로 정의하고 `dt`를 곱한다. 안 그러면 기기마다 다른 게임이 된다.
- `dt`는 반드시 상한을 둔다. 탭 복귀 시 물리가 폭발한다.

**부드러움**
- `x += (target - x) * min(1, dt * k)` — 이 한 줄이 모든 부드러운 움직임의 근원.
- `sin(t * 속도 + seed)` — 이 한 줄이 모든 살아있는 흔들림의 근원. `seed`로 개체마다 어긋나게.

**캔버스**
- DPR을 `setTransform`으로 흡수하면, 게임 코드는 DPR을 몰라도 된다.
- `save` → `translate`/`rotate`/`scale` → 로컬 좌표로 그리기 → `restore`.
- `quadraticCurveTo(꼭짓점, 다음_중점)` — 점들을 매끄럽게 잇는 정석.
- `globalAlpha`는 대입하지 말고 곱하라.

**게임 필**
- 히트박스는 플레이어에게 유리하게 거짓말한다. (피격은 작게, 획득은 크게)
- 자석, 무적 시간, 입력 디바운스 — 없으면 "이상한데 왜인지 모르겠는" 게임이 된다.
- 올라가는 음 = 좋은 일, 내려가는 음 = 나쁜 일.

**손그림**
- 결정론적 노이즈(`wob`) 없이는 형체가 유지되지 않는다.
- `Math.floor(t * 7)` 양자화 없이는 손그림이 아니라 해파리가 된다.

**React**
- 초당 60번 변하는 건 캔버스에, 가끔 변하는 것만 `setState`로.
- `useEffect`에서 시작한 것은 정리 함수에서 반드시 멈춘다.

**모바일**
- `touch-action: none`이 없으면 드래그가 끊긴다.
- 손가락은 화면을 가린다. 오프셋을 둬라.
- 오디오는 사용자 제스처 안에서만 깨울 수 있다.

## 20. 용어 사전

| 용어 | 뜻 |
|---|---|
| **델타 타임 (dt)** | 지난 프레임부터 흐른 시간(초). 모든 프레임 독립성의 기초 |
| **게임 루프** | update → draw 를 초당 60번 반복하는 구조 |
| **터널링** | 물체가 너무 빨리 움직여 충돌 판정을 건너뛰는 현상 |
| **선형 보간 (lerp)** | 두 값 사이를 비율로 섞기. `a + (b - a) * k` |
| **지수 감쇠** | 목표까지 남은 거리의 일정 비율씩 다가가기. ease-out의 정체 |
| **정규화 (normalize)** | 벡터의 길이를 1로 만들기. 방향만 남기고 크기 버리기 |
| **클램프 (clamp)** | 값을 최소/최대 범위 안에 가두기 |
| **센티널 값** | "해당 없음"을 뜻하는 특수값. 여기선 `eatT = -1` |
| **히트박스** | 충돌 판정용 영역. 보이는 그림과 일부러 다르게 만든다 |
| **i-frame (무적 시간)** | 피격 후 잠시 무적. 연타 피격 방지 |
| **DPR** | Device Pixel Ratio. CSS 픽셀 1개당 하드웨어 픽셀 수 |
| **boiling line** | 손그림 애니메이션에서 선이 프레임마다 미묘하게 달라지는 효과 |
| **fract** | 소수 부분만 취하기. `x - floor(x)` |
| **오브젝트 풀** | 객체를 버리지 않고 재사용해 GC를 피하는 기법 |
| **FSM (유한 상태 기계)** | 정해진 상태들 사이를 규칙대로 오가는 구조. `phase`가 그것 |
